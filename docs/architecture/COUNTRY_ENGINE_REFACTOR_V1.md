# Техническая спецификация универсального движка v1

Статус: **следующая реализационная задача после утверждения профиля**
Целевые страны первого сравнения: **Испания + Уругвай**

## 1. Проблема текущей реализации

Текущий `js/spain-calculator.js` не является универсальным движком:

- публичная функция называется `calculateSpain()`;
- пакет с другим `country_id` отклоняется;
- идентификаторы `ES_DNV`, `ES_NLV`, `ES_HIGHLY_QUALIFIED` и другие зашиты в код;
- проверки испанского социального страхования и отдельных испанских требований выполняются внутри функций;
- `BASIS_ROUTE` связывает ответ пользователя с одним заранее выбранным вариантом;
- `selectionScore` добавляет этому варианту `+100`, поэтому реальное сравнение всех вариантов искажается.

Следствие: если выбранный пользователем сценарий не проходит, но другой вариант полностью доступен, система может показать ложноотрицательный результат.

## 2. Цель рефакторинга

Создать единый движок:

```js
calculateCountry(profile, countryPackage, calculationContext)
```

и агрегатор:

```js
calculateCountries(profile, countryPackages, calculationContext)
```

Один и тот же профиль должен без преобразования рассчитываться для Испании и Уругвая.

## 3. Границы ответственности

### 3.1 Универсальный движок знает

- канонические статусы;
- типы правил;
- правила обработки отсутствующих данных;
- порядок оценки маршрута;
- порядок выбора лучшего варианта;
- разделение юридической и практической оценки;
- контракт результата.

### 3.2 Пакет страны знает

- варианты легализации;
- допустимые гражданства;
- места и способы подачи;
- допустимые типы дохода;
- финансовые формулы;
- правила семьи;
- работу и налоги;
- ПМЖ и гражданство;
- города, школы, животных и ЛГБТ-факты;
- источники и уровень подтверждения;
- список открытых вопросов.

### 3.3 Расчётный контекст знает

- валютные курсы;
- источник курсов;
- дату и время курсов;
- допустимую давность курса;
- дату расчёта;
- версию движка.

Пользователь не редактирует расчётный контекст.

### 3.4 Интерфейс знает

- как собрать универсальный профиль;
- как показать недостающие ответы;
- как запросить уточнения конкретного варианта после первого расчёта;
- как объяснить результат обычным языком.

Интерфейс не рассчитывает юридические статусы самостоятельно.

## 5. Контракт входа

```js
calculateCountry(profile, countryPackage, context)
```

### `profile`

Строго соответствует `user-profile-v1.schema.json`.

### `countryPackage`

Соответствует актуальной схеме исследовательского пакета. Идентификаторы вариантов могут быть любыми (`ES_DNV`, `UY_PERMANENT_RESIDENCE`), но универсальный код не содержит их строковых значений.

### `context`

Пример:

```json
{
  "calculation_date": "2026-07-18",
  "engine_version": "1.0.0",
  "fx": {
    "base_currency": "USD",
    "rates": {
      "EUR": 0.86,
      "UYU": 39.7,
      "RUB": 88.5
    },
    "source": "configured-provider",
    "as_of": "2026-07-18T09:00:00Z",
    "max_age_hours": 48
  }
}
```

Если нужного курса нет или он просрочен, движок не просит пользователя ввести курс. Он возвращает системную ошибку расчёта `CALCULATION_CONTEXT_INCOMPLETE`; страна не получает статус `UNSUITABLE`.

## 6. Нормализация профиля

Перед оценкой:

1. валидировать JSON Schema;
2. не превращать пустые значения в ноль или `false`;
3. пересчитать денежные суммы во внутреннюю базовую валюту;
4. сохранить исходную сумму, валюту, курс и дату пересчёта;
5. создать отдельный сценарий для каждого гражданства пользователя;
6. не преобразовывать `TOURIST_OR_VISA_FREE` в «резидентство есть/нет» заранее.

## 7. Оценка каждого варианта легализации

Каждый вариант оценивается независимо по полному набору применимых правил.

Пример результата правила:

```json
{
  "rule_id": "income_main_minimum",
  "scope": "ROUTE_STATUS",
  "outcome": "PASS",
  "status_effect": "SUITABLE",
  "client_field": "income.primary.monthly_provable",
  "data_field": "route_income.minimum_income_main_applicant",
  "message": "Подтверждаемый доход превышает требуемый минимум.",
  "source_fact_ids": ["ES_FACT_001"]
}
```

Допустимые исходы:

- `PASS`;
- `FAIL`;
- `CLIENT_MISSING`;
- `COUNTRY_MISSING`;
- `INDIVIDUAL_REVIEW`;
- `NOT_APPLICABLE`.

## 8. Агрегация статуса варианта

Приоритет определяется не суммой баллов, а подтверждённой природой результата:

1. `UNSUITABLE` — есть подтверждённый жёсткий блокер;
2. `INDIVIDUAL_REVIEW_REQUIRED` — результат зависит от индивидуальных документов или истории;
3. `INSUFFICIENT_COUNTRY_DATA` — отсутствует факт страны, способный изменить юридический вывод;
4. `PRELIMINARY_SUITABLE` — не хватает ответа пользователя;
5. `SUITABLE_WITH_CONDITIONS` — юридически доступно после известных действий;
6. `SUITABLE` — подтверждённых препятствий и обязательных нерешённых условий нет.

Практические проблемы города, школы, лекарства или животного не переводят юридически доступный вариант в `UNSUITABLE`.

## 9. Выбор лучшего варианта без `selectionScore +100`

### 9.1 Запрещено

- назначать вариант по ответу `income.primary.type`;
- использовать `BASIS_ROUTE`;
- давать выбранному варианту абсолютный бонус;
- скрывать доступный альтернативный вариант.

### 9.2 Лексикографический порядок

Варианты сортируются последовательно, а не по непрозрачному проценту:

1. статус варианта:
   - `SUITABLE`;
   - `SUITABLE_WITH_CONDITIONS`;
   - `PRELIMINARY_SUITABLE`;
   - `INSUFFICIENT_COUNTRY_DATA`;
   - `INDIVIDUAL_REVIEW_REQUIRED`;
   - `UNSUITABLE`;
2. закрывает ли вариант обязательную долгосрочную цель;
3. доступен ли хотя бы один приемлемый способ подачи;
4. подходит ли семья;
5. подходит ли тип и география дохода;
6. выполнен ли известный финансовый порог;
7. меньше ли неподтверждённых фактов страны;
8. меньше ли недостающих ответов пользователя;
9. меньше ли известных обязательных действий;
10. только при полном равенстве — предпочтение сценария, близкого основному источнику дохода.

Это не единый балл. В результате можно объяснить, почему один вариант выбран раньше другого.

### 9.3 `best_route` и `closest_route`

- `best_route` существует, если найден вариант со статусом выше `UNSUITABLE`;
- `closest_route` используется, когда все варианты неподходящие, и выбирается по минимальному числу подтверждённых блокеров и устранимости причин;
- интерфейс при отрицательном результате пишет «Наиболее близкий вариант», а не «Подходящий вариант».

## 10. Специфические вопросы вариантов

Основная анкета не спрашивает Seguridad Social или уругвайское доказательство средств.

После первой оценки вариант может вернуть:

```json
{
  "follow_up_questions": [
    {
      "question_id": "ES_DNV_SOCIAL_SECURITY_PLAN",
      "route_id": "ES_DNV",
      "reason": "Ответ может изменить предварительный статус варианта.",
      "allowed_answers": ["REGISTER_IN_SPAIN", "FOREIGN_CERTIFICATE", "UNKNOWN"]
    }
  ]
}
```

Ответ `UNKNOWN` допустим и не равен отказу.

## 11. Контракт результата

```json
{
  "engine_version": "1.0.0",
  "profile_schema_version": "user-profile-v1",
  "country_id": "ES",
  "country_group": "SUITABLE_WITH_CONDITIONS",
  "best_route_id": "ES_NLV",
  "closest_route_id": null,
  "routes": [],
  "legal_evaluation": {},
  "practical_evaluation": {},
  "client_missing": [],
  "country_missing": [],
  "follow_up_questions": [],
  "fx_used": [],
  "sources": []
}
```

В карточке должны быть раздельные секции:

- юридический вывод;
- лучший или наиболее близкий вариант;
- почему подходит / почему не подходит;
- что требуется сделать;
- какие ответы пользователя нужны;
- какие данные страны ещё не подтверждены;
- практическая пригодность;
- города;
- ПМЖ и гражданство с понятными пояснениями;
- использованный валютный курс и дата.

## 12. Испания после миграции

Испанский пакет подключается без логики `if (country_id === 'ES')` в ядре.

Особые требования Испании должны быть представлены:

- полями пакета;
- фактами и источниками;
- декларативными правилами;
- уточняющими вопросами варианта.

До закрытия пробела по социальному страхованию DNV система не должна категорично требовать регистрацию в Seguridad Social. До подтверждения официального порога высококвалифицированного специалиста вариант должен иметь `INSUFFICIENT_COUNTRY_DATA`.

## 13. Уругвай как вторая страна

После миграции Испании создаётся пакет Уругвая v2.2. Отдельный `uruguay-calculator.js` запрещён.

Минимальные варианты исследования:

- постоянная резиденция;
- временная резиденция;
- цифровой кочевник;
- Mercosur при подходящем втором гражданстве;
- другие релевантные варианты, подтверждённые исследованием.

Один вызов `calculateCountries()` должен вернуть сопоставимые карточки Испании и Уругвая.

## 14. Обязательные тесты

1. Выбранный пользователем тип дохода не назначает вариант-победитель.
2. DNV не подходит, но другой испанский вариант подходит — выбирается другой вариант.
3. Один ребёнок остаётся ребёнком в расчёте и интерфейсе.
4. Туристический статус отличается от ВНЖ и гражданства.
5. Пользователь не вводит валютный курс.
6. Отсутствие курса даёт системную ошибку, а не `UNSUITABLE`.
7. Отсутствующий факт страны даёт `INSUFFICIENT_COUNTRY_DATA`.
8. Отсутствующий ответ пользователя даёт `PRELIMINARY_SUITABLE`.
9. Особые обстоятельства дают `INDIVIDUAL_REVIEW_REQUIRED`.
10. Неподходящий бюджет города не меняет юридический статус варианта.
11. Один профиль рассчитывается для `ES` и `UY` без изменения структуры.
12. В файлах универсального ядра нет идентификаторов `ES_` и `UY_`.
13. Все существующие подтверждённые тесты Испании продолжают проходить либо заменены эквивалентными тестами нового контракта.

## 15. Порядок миграции кода

1. Добавить строгую схему профиля и фикстуры.
2. Вынести общие статусы и функции нормализации.
3. Реализовать универсальную оценку одного правила.
4. Реализовать оценку одного варианта.
5. Реализовать агрегацию статуса.
6. Реализовать лексикографический выбор лучшего варианта.
7. Перенести Испанию на новое ядро.
8. Оставить `calculateSpain()` как временную обёртку, вызывающую `calculateCountry()`.
9. Перевести интерфейс на универсальный профиль.
10. После стабилизации удалить старую испанскую реализацию.
11. Подключить пакет Уругвая.

## 16. Условия принятия рефакторинга

Рефакторинг принят, когда:

- текущий пилот Испании даёт те же подтверждённые юридические результаты без `BASIS_ROUTE`;
- тест альтернативного подходящего варианта проходит;
- в основной анкете нет ручного курса и вопросов конкретной программы;
- Испания и Уругвай обрабатываются одной функцией;
- юридический и практический выводы остаются разделёнными;
- все выводы содержат путь к правилу, факту и источнику;
- старый корневой рейтинг не используется как основной пользовательский интерфейс.
